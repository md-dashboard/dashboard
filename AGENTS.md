# Codex 작업 지침

## 프로젝트 개요

이 저장소는 여러 쇼핑몰의 주문/배송 엑셀 파일을 하나의 공통 스키마로 표준화해 보여주는 React 대시보드입니다. 현재 구현된 범위는 CJ몰, 네이버, GS몰 원본 엑셀 업로드, 소스 자동 탐지, 컬럼 매핑, 중복 주문 upsert, 통합 테이블 검색/필터링입니다.

사용자가 만들려는 큰 방향은 사이트마다 다른 주문/송장 데이터를 표준화하고, 출고사나 택배사에 맞는 양식으로 다시 가공해 전달할 수 있는 운영 대시보드입니다. 향후에는 로젠/CJ/한진 등 택배사별 송장 양식, 오늘 출고 건수, 판매상태별 집계, 취소/반품 구분, 상품/회사별 표시, 순익 계산 같은 기능이 추가될 가능성이 큽니다.

## 기술 스택

- Vite + React + TypeScript
- React 19
- SheetJS `xlsx`로 브라우저에서 엑셀 파일 파싱
- Oxlint로 린트
- 현재 백엔드, DB, 영구 저장소는 없습니다. 새로고침하면 병합 데이터가 초기화됩니다.

## 주요 명령어

- 개발 서버: `npm run dev`
- 프로덕션 빌드: `npm run build`
- 린트: `npm run lint`
- 빌드 결과 미리보기: `npm run preview`

작업 후에는 최소 `npm run build`와 `npm run lint`를 실행해 타입/번들/린트 상태를 확인하세요.

## 코드 구조

- `src/App.tsx`
  - 단일 페이지 UI입니다.
  - 파일 드래그/클릭 업로드, 소스 선택 확인, 병합 결과 테이블, 검색, 쇼핑몰별 필터, 업로드 이력을 관리합니다.
  - 병합 데이터는 `Map<string, Row>` 상태로 들고 있으며 `_key` 기준으로 upsert합니다.

- `src/schema.ts`
  - 도메인 표준화 규칙의 중심 파일입니다.
  - `SourceType`, 쇼핑몰 라벨, 자동 탐지용 `FINGERPRINTS`, 원본 컬럼명 -> 공통 컬럼명 매핑, 최종 컬럼 순서, row key 구성 필드를 정의합니다.
  - 새 쇼핑몰이나 새 컬럼을 추가할 때 가장 먼저 확인해야 하는 파일입니다.

- `src/merge.ts`
  - 엑셀 파싱, 쇼핑몰 자동 탐지, 공통 행 변환, key 생성, upsert 로직을 담당합니다.
  - 엑셀 날짜 셀은 `cell.w` 표시 문자열을 우선 보존합니다. 날짜/시간을 숫자 serial이나 JS Date로 성급히 바꾸면 원본 화면 값과 달라질 수 있습니다.
  - 주문번호가 비어 있는 행은 합계/푸터 가능성이 높아 제외합니다.

- `src/App.css`, `src/index.css`
  - 운영툴 느낌의 밀도 높은 레이아웃과 디자인 토큰을 정의합니다.
  - 현재 화면은 카드형 업로드/결과 영역, 좌측 메뉴, 상단바, 고정 헤더 테이블 구조입니다.

## 데이터 모델과 규칙

- 소스는 현재 `CJ`, `NAVER`, `GS` 세 가지입니다.
- 공통 출력 컬럼은 `ALL_COLUMNS` 순서를 따릅니다.
- 내부 중복 판정 key는 `출처 + 주문번호 + 상품주문번호(아이템)` 조합입니다.
- 쇼핑몰별로 같은 의미라도 원본 컬럼명이 다르므로, 매핑 변경은 반드시 샘플 엑셀 헤더 기준으로 검증해야 합니다.
- `출고지`는 네이버 전용 컬럼이 아니라 공통 출력 컬럼입니다.
- `AUDIT_ONLY` 컬럼은 공통 필드로 합치지 않고 소스 전용 원본 값으로 보존하는 영역입니다. 현재는 활성화된 감사 전용 출력 컬럼이 없습니다. 의미 충돌 가능성이 있는 컬럼은 억지로 공통화하지 말고 이 영역으로 남기는 쪽을 우선 검토하세요.
- 새 소스를 추가할 때는 `SourceType`, `SOURCE_LABEL`, `FINGERPRINTS`, 개별 `MAP_*`, `MAPS`, 필요 시 `AUDIT_ONLY`, `SOURCES` 배열을 함께 갱신해야 합니다.

## 구현 시 주의사항

- 표준화 규칙은 UI 컴포넌트에 직접 흩뿌리지 말고 `schema.ts`에 모으세요.
- 엑셀 파싱/변환 로직은 `merge.ts`에 두고, `App.tsx`는 사용자 상호작용과 화면 상태에 집중시키세요.
- 주문/송장/정산 데이터는 운영상 민감할 수 있으므로 콘솔 로그나 테스트 fixture에 실제 개인정보를 남기지 마세요.
- 파일 업로드는 현재 브라우저 메모리에서만 처리합니다. 영구 저장, 서버 전송, 외부 API 연동을 추가할 때는 개인정보 처리 범위와 보관 정책을 먼저 정해야 합니다.
- 대량 엑셀을 다룰 가능성이 있으므로 테이블 렌더링, 검색, 변환 로직을 확장할 때 성능을 함께 고려하세요.
- 현재 `README.md`는 Vite 기본 문서에 가깝습니다. 사용자용 문서를 만들 때는 실제 주문 통합 도구 기준으로 다시 작성하는 것이 좋습니다.

## 향후 기능 후보

- 병합 결과 엑셀 다운로드
- 택배사별 송장 업로드/출고 지시 양식 변환
- 오늘 출고/배송/취소/반품 건수 집계
- 회사/상품/판매채널별 매출, 공급가, 순익 계산
- 업로드 데이터 저장 및 재방문 시 복원
- 원본 파일별 오류/누락 컬럼 리포트
- 쇼핑몰별 매핑 관리 UI

## 검증 관점

- 샘플 엑셀을 추가하거나 매핑을 바꾼 경우:
  - 자동 탐지가 기대한 소스를 맞추는지 확인
  - 주문번호가 비어 있는 합계 행이 제외되는지 확인
  - 날짜/시간 표시가 원본 엑셀 화면과 다르지 않은지 확인
  - 같은 주문을 다시 업로드했을 때 신규가 아니라 갱신으로 집계되는지 확인
  - `ALL_COLUMNS` 순서가 다운로드/테이블 요구사항과 맞는지 확인

## 협업 메모

- 사용자는 한국어로 작업 맥락을 설명하고 있으며, 답변과 문서도 한국어를 기본으로 유지하는 것이 좋습니다.
- “대시보드”라고 표현하더라도 현재 1차 목표는 시각화보다 주문 엑셀 표준화와 출고 데이터 가공에 가깝습니다.
- 기능을 추가할 때는 먼저 실제 업체/쇼핑몰/택배사 샘플 엑셀 헤더와 기대 출력 양식을 기준으로 스키마를 확정하세요.
